## 1. Главные выводы аудита (что важнее, чем в доках)

Аудит сверил каждое утверждение доков с реальным кодом. Три находки меняют расстановку приоритетов:

1. **Зрелый движок расстановки подключён НЕ туда.** Детерминированный движок (`GenerateArrangementTool` + `CollisionService` + `DesignNorms`) реален и качественен, но работает **только в чат-советнике** (через `ToolRegistry`/`SkillIndex`). На экране **AR-расстановки его нет** — там координаты мебели **придумывает сам LLM** (`AgentOrchestrator` → `PromptBuilder.swift:85` явно просит у модели `position {x,y,z}`), а коллизии лишь валидируются с **повторным запросом к LLM** (до 2 раз). Отсюда медленная и нестабильная генерация. → **Главный технический актив есть, его нужно переключить на нужный путь**, а не строить заново.

2. **«Путь без LiDAR» хуже, чем заглушка.** Кнопка «Ввести вручную» ведёт в пустоту (`.manualEntryTapped` → `.none`), формы ввода размеров нет. Хуже: **рантайм-проверки наличия LiDAR нет вообще** — единственная развилка `#if targetEnvironment(simulator)`. На реальном iPhone **без** LiDAR безусловно запускается RoomPlan и **молча не даёт результата** — пользователь в тупике без объяснения. Это отсекает массовый рынок РФ (**разрыв №1**).

3. **Монетизации в коде нет вообще.** Сквозной поиск по всему репозиторию: `ЮKassa / промокод / paywall / CPA / подписка` — **ноль** в коде (только в документах и в отдельном Next.js-демо в долларах). Единственные «денежные» следы — **мёртвый** флаг `hasSubscription` (объявлен, нигде не читается) и заблокированный в MVP `confirm_purchase_order` (always-DENY). Весь контур дохода строится с нуля.

Дополнительно подтверждено: ~~**нет каталога USDZ** (0 моделей в бандле → рендерится placeholder-бокс)~~ *(✅ устарело 2026-06-16: в бандле 8 CC0-USDZ с PBR, AR-расстановка работает — см. §9, 8–9 сессии)*; ~~**бэкенд скрейпит WB/Ozon** (Apify) вместо партнёров~~ *(✅ решено 2026-06-10: Apify-путь удалён, только партнёрский каталог)*; **«нативный вектор YDB»** на деле — `scan(limit 500)` + косинус в Node (нарушает собственное правило §9.2); **гибрид AI Lite+Pro не реализован** (всё уходит на дорогую Pro-модель); **фаза 10 сделана наполовину** (чат сохраняется, проекты комнат — нет, и хранилище в `Caches`, который ОС может вычистить).

## 3. Принципы апгрейда

1. **Сохраняем логику.** Ядро (`AgentLoop`, `ToolRegistry`, `SkillIndex`, `Providers/*`, `AIProviderRouter`, `CircuitBreaker`, `yandexgpt.js`, `gigachat.js`, `triplex-fallback.js`) — только bugfix/добавления. Всё новое строим в фичах/новых модулях.
2. **Сначала функции и деньги, дизайн потом.** Визуальный редизайн (Apple-стиль из `demo-design-mg`) — **вне этого апгрейда** (см. §7). Новый стиль пока живёт для веб-лендинга.
3. **Бэкенд no-deps** на горячем пути AI (только `fetch`). npm-библиотеки (если нужны для GLB→USDZ) — только в **изолированном** сервисе конвертации.
4. **Переиспользуем, не дублируем.** AR-движок, `CollisionDetector`, `DesignNorms`, `RoomGeometry.init`, `runCollisionCheck` уже есть — подключаем их, а не пишем заново.
5. **Регуляторика — на входе, не на выходе.** Apple anti-steering (3.1.1), 152-ФЗ, ЗоЗПП учитываем в дизайне каждой денежной фичи (см. §6).

## 6. Риски и зависимости

1. **CPA-атрибуция (C2)** — главный риск бизнес-модели: длинный цикл покупки мебели, слабые CRM фабрик «теряют» продажи, договоры/сверка. Без неё поток №2 (главная прибыль) не работает.
2. **Cold start у партнёров** — без каталога нет CPA и нет реальных USDZ. Нужен нетворкинг и бесплатная/льготная оцифровка первым фабрикам. Фичефлаг «партнёры vs Apify» (B2) — страховка.
3. **GLB→USDZ привязан к macOS** — не запускается в Cloud Function. Решается batch-конвертацией на Mac mini (`usdzconvert`); оплата зарубежного macOS-хостинга из РФ затруднена.
4. **Векторный поиск YDB (B5)** — KNN/YQL может быть недоступен через текущий Document API (DynamoDB-совместимый) → сначала PoC, иначе SQL-пре-фильтр. Не нарушить no-deps.
5. **Apple anti-steering (3.1.1)** — пронизывает A3, C1.5, C2: оплату и deeplink проектировать аккуратно, иначе отклонение ревью.
6. **Реальный marketplace API — пока mock** (`SearchMarketplaceFurnitureTool.performRealSearch` = TODO): для шага 4 (C1) и BNPL нужен рабочий backend-поиск.
7. **Рассинхрон систем координат** (A2): LLM-плоскость (угол 0,0,0) vs simd-центр в `CollisionService` — нужны тесты при переключении на движок.
